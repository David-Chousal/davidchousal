<h1>Close the loop between ears, fingers, and notation</h1>

<p>
  The app targets the awkward middle ground after you solo on a tune: you have audio, but not yet a
  structured record of what you played. The flow is deliberately practice-shaped: open or start a session,
  use <strong>Mark Bar 1</strong> to align the take with the chart, record with guardrails against
  feedback, then review transcription output before saving. Session files live under Application Support
  (<code>Application Support/TranscribeJazz/Sessions/&lt;session-id&gt;/</code>), not ephemeral temp
  storage.
</p>

<p>
  iReal Pro is the bridge to real-book changes and search; the codebase also routes MusicXML, MIDI, and
  audio through an import path built for sandboxed, security-scoped URLs.
</p>

<h2>Layered modules, one product story</h2>

<p>
  The repository mirrors a small clean-ish slice: domain models stay free of Apple frameworks; Apple-bound
  work sits in persistence, services, and SwiftUI features.
</p>

<ul>
  <li>
    <strong>App</strong> — Entry (<code>TranscribeJazzApp</code>), app delegate hooks, root navigation.
  </li>
  <li>
    <strong>Domain</strong> — Pure models: sessions, transcriptions, charts, audio assets, shared enums.
  </li>
  <li>
    <strong>Persistence</strong> — Core Data stack, entities, repository, and file-backed session store.
  </li>
  <li>
    <strong>Services</strong> — iReal linker, import coordinator, UTType helpers, audio session management,
    recorder and player engines, onset/pitch pipeline, swing quantizer, tab mapper, transcription store.
  </li>
  <li>
    <strong>Features</strong> — SwiftUI surfaces: Library, New Session, Practice Mode, Review/Edit, Import
    Center.
  </li>
  <li>
    <strong>Utilities</strong> — Errors, logging, formatters, app constants.
  </li>
</ul>

<h2>What the app demonstrates today</h2>

<h3 class="!mt-8 !mb-3 text-base font-semibold text-case-ink">Practice mode</h3>

<ul>
  <li>
    <strong>Speaker guard</strong> — Recording is blocked when output is the built-in speaker, reducing
    mic feedback on kitchen-table practice.
  </li>
  <li>
    <strong>Durable paths</strong> — Audio is written to Application Support, not <code>/tmp</code>.
  </li>
  <li>
    <strong>Mark Bar 1</strong> — Explicit control to align the first bar of the take with the chart.
  </li>
  <li>
    <strong>Input metering</strong> — Real-time level feedback while recording.
  </li>
</ul>

<h3 class="!mt-8 !mb-3 text-base font-semibold text-case-ink">Transcription and review</h3>

  <li>

    lightweight migration.
  </li>

<CaseStudySection id="import" kicker="Import and integrations">

<h2>Charts, files, and iReal handoff</h2>

<ul>
  <li>
    <strong>File formats</strong> — MusicXML, MIDI, and audio imports share coordinator and type
    plumbing.
  </li>
  <li>
    <strong>iReal Pro</strong> — Deep links and queries use <code>irealb</code> and <code>irealbook</code>
    URL schemes so the app can sit next to the catalog players already use.
  </li>
  <li>
    <strong>Sandbox-friendly imports</strong> — Security-scoped URL handling for user-picked files.
  </li>
</ul>

<p>
  Info.plist expectations include microphone usage copy and <code>LSApplicationQueriesSchemes</code> for
  those schemes — details live in the repo’s setup notes.
</p>

<CaseStudySection id="roadmap" kicker="Roadmap">

<h2>Phased delivery (from the repo README)</h2>

<ul>
  <li>
    <strong>Phase 1 — Functional MVP</strong> — Library and new session flows; open iReal search; practice
    recording with Mark Bar 1; basic per-measure transcription list; review for rhythm simplification and
    swing; save sessions and files.
  </li>
  <li>
    <strong>Phase 2 — Import power</strong> — MusicXML plus audio import; form and measure numbers from
    MusicXML; backing-track playback.
  </li>
  <li>
    <strong>Phase 3 — Polish</strong> — Stronger notation/tab rendering; faster editing (e.g. drag note
    timing, quick ties); export formats.
  </li>
</ul>

<CaseStudySection id="setup" kicker="Setup">

<h2>Building and extending</h2>

<p>
  The Xcode target may still use a legacy app entry (<code>T_Jazz_ProApp.swift</code>); the README
  describes either swapping in <code>TranscribeJazzApp.swift</code> or wiring the new root view from the
  existing entry. New source folders need to be added to the target; <code>CoreDataModels.xcdatamodeld</code> must be included in the build.
</p>

<p>
  For permissions and iReal discovery, add <code>NSMicrophoneUsageDescription</code> and
  <code>LSApplicationQueriesSchemes</code> entries (<code>irealb</code>, <code>irealbook</code>) via
  generated Info.plist settings or the bundled <code>Resources/Info.plist</code> as documented in
  <strong>SETUP.md</strong> on GitHub.
</p>
