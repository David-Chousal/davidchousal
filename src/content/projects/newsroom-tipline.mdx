<h1>Let readers reach the desk without unnecessary exposure</h1>

<p>
  The system is built for a classic newsroom split: a <strong>public</strong> tip line where readers
  submit a message and optional contact signal, and an <strong>internal</strong> moderation surface where
  editors review, tag, approve or reject, and leave an accountable trail. Contact material is never stored
  as reversible plaintext in the database path documented in the repo — only a hash plus explicit consent
  — so follow-up stays intentional rather than accidental bulk PII retention.
</p>

<p>
  The codebase is a <strong>pnpm workspace</strong> aimed at TypeScript end-to-end: a Vite React app for
  the web UI and a Fastify API with Prisma on PostgreSQL, with Docker and Google Cloud Run + Cloud SQL
  called out for production-shaped deployment.
</p>

</CaseStudySection>

<CaseStudySection id="product" kicker="Product surface">

<h2>What ships in scope</h2>

<ul>
  <li>
    <strong>Public tip submission</strong> — Message field (length bounds per schema), optional contact
    channel, explicit consent, rate limiting on the public route.
  </li>
  <li>
    <strong>Moderation queue</strong> — List and filter tips (e.g. by <code>pending</code> status); patch
    workflow for approve / reject / tag-style updates as implemented in the API.
  </li>
  <li>
    <strong>Audit trail</strong> — Each tip carries a structured audit history (actor, action, timestamp) so
    newsroom decisions are defensible after the fact.
  </li>
  <li>
    <strong>Accessibility posture</strong> — WCAG 2.1 AA stated target; keyboard navigation and screen
    reader support; <code>jsx-a11y</code> in CI per README.
  </li>
  <li>
    <strong>Operational tasks</strong> — Internal cron-style endpoints (shared secret) for feed ingestion
    and for purging old rejected tips on a schedule.
  </li>
</ul>

</CaseStudySection>

<CaseStudySection id="security" kicker="Security and privacy">

<h2>Threats the README calls out explicitly</h2>

<ul>
  <li>
    <strong>PII minimization</strong> — Contact information is hashed (SHA-256) before persistence; raw
    email/phone is not stored.
  </li>
  <li>
    <strong>Consent</strong> — Boolean consent stored with the tip when the reader agrees to follow-up.
  </li>
  <li>
    <strong>Abuse</strong> — Public <code>POST /api/tips</code> is rate-limited (documented as 10 requests
    per 15 minutes); honeypot field on the form for bot noise.
  </li>
  <li>
    <strong>Retention</strong> — Rejected tips can be auto-purged after 90 days (configurable), reducing
    long-tail liability.
  </li>
  <li>
    <strong>Transport and browser policy</strong> — CORS locked to production domains in the documented
    deployment story.
  </li>
  <li>
    <strong>Authentication gap (honest)</strong> — README lists moderation auth as <strong>TODO</strong>:
    OAuth provider, JWT middleware, protected routes, and reviewer IDs fully wired into audit metadata are
    the next hardening layer.
  </li>
</ul>

<CaseStudySection id="api" kicker="API and data">

<h2>Endpoints and tip model (from repo docs)</h2>

<h3 class="!mt-8 !mb-3 text-base font-semibold text-case-ink">Public</h3>

<ul>
  <li>
    <code>POST /api/tips</code> — Submit a tip (rate-limited).
  </li>
  <li>
    <code>GET /healthz</code> — Health check for orchestration.
  </li>
</ul>

<h3 class="!mt-8 !mb-3 text-base font-semibold text-case-ink">Protected (moderation; auth per roadmap)</h3>

<ul>
  <li>
    <code>GET /api/tips?status=pending</code> — Filtered listing.
  </li>
  <li>
    <code>PATCH /api/tips/:id</code> — Update status, tags, and related moderation fields.
  </li>
</ul>

<h3 class="!mt-8 !mb-3 text-base font-semibold text-case-ink">Internal (cron)</h3>

<ul>
  <li>
    <code>POST /tasks/ingest</code> — Feed ingestion (e.g. “Live Results” style pipelines).
  </li>
  <li>
    <code>POST /tasks/purge-rejected</code> — Purge stale rejected tips.
  </li>
</ul>

<p>
  The documented <strong>Tip</strong> entity includes CUID <code>id</code>, timestamps, bounded <code>message</code>, <code>contactHash</code>, <code>consent</code>, <code>status</code> (<code>pending</code> | <code>approved</code> | <code>rejected</code>), string <code>tags</code>, optional <code>reviewerId</code> once auth lands, and the <code>audit</code> JSON array.
</p>

</CaseStudySection>

<CaseStudySection id="quality-deploy" kicker="Quality and deployment">

<h2>Tests, a11y, and GCP</h2>

<ul>
  <li>
    <strong>Unit / API tests</strong> — Vitest from workspace root (<code>pnpm test</code>).
  </li>
  <li>
    <strong>E2E</strong> — Playwright via <code>pnpm --filter @tipline/web test:e2e</code> (package name as in
    README).
  </li>
  <li>
    <strong>Accessibility</strong> — Automated linting plus manual keyboard and screen reader passes as
    recommended in docs.
  </li>
  <li>
    <strong>GCP deployment outline</strong> — Cloud SQL Postgres, Artifact Registry / GCR image push, two
    Cloud Run services (API + static/SSR web image per Dockerfile), secrets in Secret Manager, and
    Scheduler invoking purge with an <code>x-cron-secret</code> header pattern.
  </li>
</ul>

</CaseStudySection>

<CaseStudySection id="setup" kicker="Setup">

<h2>Local development</h2>

<p>
  Prerequisites: <strong>Node.js 18+</strong>, <strong>pnpm 9+</strong>, and PostgreSQL 14+ (local,
  Docker, or a hosted instance — <strong>SETUP_POSTGRES.md</strong> documents low-friction options). Clone <a href="https://github.com/David-Chousal/newsroom-tipline-webserver-front-and-back-end">the repository</a>, run <code>pnpm install</code>, copy <code>apps/api/.env.example</code> to <code>apps/api/.env</code> with
  a valid <code>DATABASE_URL</code>, then from <code>apps/api</code> run Prisma generate and migrate. From
  the repo root, <code>pnpm dev</code> starts the Vite frontend on port <strong>3000</strong> and the API
  on <strong>3001</strong> as documented in <strong>QUICKSTART.md</strong>.
</p>

<p>
  For editorial framing and risk language aligned to newsroom ethics, see <strong>EDITORIAL_RATIONALE.md</strong>; for container-based workflows, <strong>DOCKER_SETUP.md</strong> and <strong>PROJECT_SUMMARY.md</strong> round out implementation detail on GitHub.
</p>
