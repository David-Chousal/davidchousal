<h1>A complete issue-hold-verify loop for privacy-preserving credentials</h1>

<p>
  This project shows how verifiable credentials can work end-to-end on Apple platforms without turning the
  holder into a passive data hose. An issuer provisions a credential, the holder stores it locally and
  decides what to show, and a verifier requests specific attributes and checks authenticity — including
  proof that the presenting device controls the credential and that the session has not been replayed.
</p>

<p>
  Transport between phones is intentionally simple: JSON payloads moved by QR codes, so the demo stays
  easy to run and audit without standing up a bespoke transport layer.
</p>

<CaseStudySection id="architecture" kicker="Architecture">

<h2>Four moving parts, one protocol story</h2>

<p>
  The repository is organized as a small system rather than a single app. Each piece has a narrow job and
  shares types and crypto where it matters.
</p>

<ul>
  <li>
    <strong>IssuerService (Node.js / Express)</strong> — Issues credentials and exposes the HTTP surface
    the demo uses to provision holders.
  </li>
  <li>
    <strong>HolderApp (iOS)</strong> — Stores credentials with device-bound keys, gates sensitive actions
    with biometrics, and builds presentations when the user agrees.
  </li>
  <li>
    <strong>VerifierApp (iOS)</strong> — Encodes verification requests (including session nonces), scans
    the holder’s response QR, and runs the validation pipeline.
  </li>
  <li>
    <strong>Shared (Swift package)</strong> — Holds shared models and cryptographic helpers so holder and
    verifier implementations do not drift.
  </li>
</ul>

<CaseStudySection id="features" kicker="Features and highlights">

<h2>What the stack demonstrates</h2>

<ul>
  <li>
    <strong>Selective disclosure</strong> — SD-JWT-style presentations: only claims the verifier asked for
    leave the holder device, backed by salted-hash disclosure paths.
  </li>
  <li>
    <strong>Proof of possession</strong> — ECDSA P-256 signatures with device-bound keys so possession is
    cryptographic, not just “I have a JPEG of a credential.”
  </li>
  <li>
    <strong>Replay protection</strong> — Session-specific nonces bind a presentation to a single verifier
    request.
  </li>
  <li>
    <strong>Revocation</strong> — Status List 2021–style status checks so credentials can be invalidated
    before natural expiry.
  </li>
  <li>
    <strong>Biometric consent</strong> — Face ID / Touch ID gates high-risk actions so sharing is an
    explicit, user-paced step.
  </li>
  <li>
    <strong>Secure storage</strong> — Keychain with device-only access for key material appropriate to a
    wallet-shaped PoC.
  </li>
  <li>
    <strong>QR-mediated exchange</strong> — Request and response payloads as JSON in QR codes for
    demonstrable, offline-friendly handoff between apps.
  </li>
  <li>
    <strong>Verifier-side validation</strong> — Issuer signature verification, disclosure integrity,
    freshness windows, and revocation are composed into a fail-closed path.
  </li>
</ul>

</CaseStudySection>

<CaseStudySection id="selective-disclosure" kicker="Selective disclosure">

<h2>Share the minimum viable proof</h2>

<p>
  Verifiers rarely need the full credential graph. The holder flow is built around disclosing individual
  claims and assembling only the hashes and disclosures a policy requires — aligned with SD-JWT-style
  presentation patterns so auditors can map behavior to emerging standards language.
</p>

<CaseStudySection id="security" kicker="Security and trust model">

<h2>Biometrics at the gate, cryptography under the hood</h2>

<p>
  Sensitive actions sit behind the device trust model users already understand (Face ID / Touch ID). Under
  that surface, presentations are time-bounded (<code>iat</code> / <code>exp</code>), tied to verifier
  nonces, and signed with holder keys so verifiers get both issuer authenticity and proof of possession.
</p>

<p>
  Validation is deliberately conservative: issuer signatures and disclosure hashes detect tampering;
  freshness and nonce checks limit stale or copied payloads; revocation status is consulted so issuance
  mistakes or policy changes can be enforced before expiry. The demo posture treats privacy as a product
  constraint — minimal logging and no tracking narrative in the reference documentation.
</p>

</CaseStudySection>
