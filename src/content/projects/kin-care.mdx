<h1>Building a calmer operating system for caregiving</h1>

<p>
  Kin is the mobile surface for shared care: one place for the household and invited helpers to see what
  matters today — medications, tasks, visits, and lightweight health context — without defaulting to a
  parallel chat app for every handoff.
</p>

<p>
  The shipping codebase is an <strong>Expo</strong> app (<code>slug</code> <code>kin-care</code>, deep link
  scheme <code>kincare</code>) talking to <strong>Supabase</strong> for auth, Postgres, and realtime
  channels. Auth sessions can complete from email magic links via <code>expo-linking</code> by parsing
  Supabase hash tokens on cold start and URL events.
</p>

<h2>Client, backend, and automation in one repo</h2>

<h3 class="!mt-8 !mb-3 text-base font-semibold text-case-ink">Mobile client</h3>

<ul>
  <li>
    <strong>Expo SDK 54 / React Native 0.81</strong> — Dev client, native modules for notifications, secure
    store, image picker, localization, and more (<code>app.json</code> + config plugins).
  </li>
  <li>
    <strong>Navigation</strong> — <code>@react-navigation</code> native stack; separate auth and main
    stacks, post-auth onboarding gate, theme tokens wired to system / user preference.
  </li>
  <li>
    <strong>Data layer</strong> — <strong>TanStack Query</strong> with <code>offlineFirst</code> network
    mode for queries and mutations; network-status integration for resilient retries.
  </li>
  <li>
    <strong>State</strong> — <strong>Zustand</strong> for auth and theme stores among others.
  </li>
  <li>
    <strong>Observability</strong> — <strong>Sentry</strong> initialized at startup; React Query cache /
    mutation errors forwarded for triage.
  </li>
  <li>
    <strong>i18n</strong> — <code>i18next</code> / <code>react-i18next</code> with language sync hooks
    (RTL phase-2 notes live in repo docs).
  </li>
</ul>

<CaseStudySection id="reliability" kicker="Sync and trust">

<h2>Realtime, offline-first, and reminders</h2>

<ul>
  <li>
    <strong>Realtime sync</strong> — Dedicated hook wires Supabase realtime into the client data model so
    multiple caregivers see updates without manual refresh.
  </li>
  <li>
    <strong>Medication dose reminders</strong> — Sync hook aligns local notification scheduling with
    server-backed medication state.
  </li>
  <li>
    <strong>Connectivity UX</strong> — A connectivity banner surfaces degraded network state alongside
    offline-first queries.
  </li>
  <li>
    <strong>Trust posture</strong> — Errors surface to Sentry; mutations and queries default to cautious
    network modes so the “happy path” stays honest when connectivity drops.
  </li>
</ul>

</CaseStudySection>

<CaseStudySection id="subscriptions" kicker="Subscriptions">

<h2>B2C IAP and Supabase as source of truth</h2>

<p>
  The repo documents a <strong>B2C subscription MVP</strong>: consumer tiers only, with RevenueCat as the
  mobile payment rail and Supabase storing derived entitlement state (including operator overrides for demos
  or support).
</p>

<ul>
  <li>
    <strong>Tiers called out in docs</strong> — <strong>Core</strong> (free / display naming), <strong>Family</strong> ($4.99/mo), <strong>Care Team</strong> ($8.99/mo); internal <code>effective_tier</code> resolution is centralized for gating (including LLM caps on paid plans).
  </li>
  <li>
    <strong>Webhook path</strong> — <code>rc-webhook</code> processes RevenueCat events into cache tables;
    enterprise or agency billing is explicitly deferred to future epics in the same doc set.
  </li>
</ul>

<p>
  Store privacy, data safety, and go/no-go templates live under <code>docs/</code> for release discipline.
</p>

<h2>Running and shipping from the repo</h2>

<p>
  Clone <a href="https://github.com/David-Chousal/kin.care">github.com/David-Chousal/kin.care</a>, copy{" "}
  <code>.env.example</code> to a local env file with Supabase and third-party keys, then <code>npm install</code>{" "}
  and <code>npx expo start</code> (or platform run scripts). Supabase schema and functions apply from the{" "}
  <code>supabase/</code> directory using the normal Supabase CLI workflow.
</p>

<p>
  Production-style builds use <strong>EAS</strong> (<code>eas:build:preview</code>,{" "}
  <code>eas:build:production</code>, <code>eas:submit:production</code>); legal pages can be regenerated via{" "}
  <code>npm run build:legal</code>. Operational checklists (Sentry env verification, store readiness,
  subscription QA) are in <strong>docs/</strong> — see the related link cards above.
</p>
